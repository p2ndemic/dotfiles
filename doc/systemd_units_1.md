1. ВВЕДЕНИЕ: ЧТО ТАКОЕ SYSTEMD ЮНИТЫ?
================================================================================
Systemd — это инициализатор и менеджер сервисов в Linux. 
Юнит (Unit) — это единица конфигурации. Файлы сервисов обычно лежат:
- Системные: /etc/systemd/system/  или  /usr/lib/systemd/system/
- Пользовательские: ~/.config/systemd/user/  или  /usr/lib/systemd/user/

Типы юнитов:
- .service : Демон/программа (о чем этот конспект).
- .target  : Группа сервисов (аналог runlevel).
- .timer   : Аналог cron.
- .oneshot : Задача, которая выполняется один раз и завершается.

================================================================================
2. СТРУКТУРА ФАЙЛА СЕРВИСА
================================================================================
Файл состоит из трех основных секций:

[Unit]
  Описание, зависимости, условия запуска.
  Ключевые директивы:
  - Description=   : Человекочитаемое имя.
  - After=         : Запускать ПОСЛЕ указанных юнитов (порядок, не зависимость).
  - PartOf=        : Если указанный юнит останавливается, этот тоже стопается.
  - Condition...   : Проверка условий перед запуском (если false — сервис skipped).

[Service]
  Как запускать процесс.
  Ключевые директивы:
  - Type=          : simple, exec, forking, oneshot, notify.
  - ExecStart=     : Команда запуска.
  - ExecReload=    : Команда перезагрузки конфига без рестарта.
  - Restart=       : on-failure, always, on-abnormal.
  - Slice=         : Группа ресурсов (cgroups) для контроля памяти/CPU.
  - Environment=   : Переменные окружения.

[Install]
  Как включать сервис в автозагрузку.
  Ключевые директивы:
  - WantedBy=      : В какой target добавить ссылку при команде 'enable'.

3. ВАЖНЫЙ ВОПРОС: ЧТО БУДЕТ, ЕСЛИ НЕ УКАЗАТЬ [Install]?
================================================================================
Если секция [Install] отсутствует:

1. Команда 'systemctl enable <service>' НЕ сработает.
   Вы получите ошибку: "Unit file ... does not contain installation info".
   
2. Сервис НЕ будет запускаться автоматически при загрузке системы 
   или при входе пользователя.

3. Сервис МОЖНО запустить вручную: 'systemctl start <service>'.
   Он будет работать, пока вы его не остановите или не перезагрузитесь.

4. Использование: Такие юниты часто используются как шаблоны (templates) 
   или запускаются как зависимости других сервисов, но не как точка входа.

4. СЦЕНАРИИ ДЛЯ ГРАФИЧЕСКОЙ СЕССИИ (WAYLAND)
================================================================================
Здесь мы рассматриваем 4 подхода к запуску приложений (mako, waybar) 
в зависимости от окружения.

--- ВАРИАНТ 1: Жесткая привязка к конкретным таргетам сессии ---
(Пример: mako)
Плюсы: Точный контроль. Сервис умрет, если умрет конкретная сессия.
Минусы: Нужно много дублирования строк для каждого композитора.

[Unit]
PartOf=wayland-session@sway.desktop.target
After=wayland-session@sway.desktop.target
# ... повторяется для labwc, niri и т.д.

[Service]
Slice=background-graphical.slice  # Слайс для фоновых граф. процессов

[Install]
WantedBy=wayland-session@sway.desktop.target
# ... повторяется для всех

--- ВАРИАНТ 2: Графические приложения со Slice и ExecReload ---
(Пример: waybar)
Отличие от Варианта 1: 
- Slice=app-graphical.slice (для интерактивных приложений).
- ExecReload=kill -SIGUSR2 $MAINPID (умная перезагрузка конфига).

--- ВАРИАНТ 3: Универсальный запуск + Whitelist (Белый список) ---
(Пример: mako через ConditionEnvironment)
Плюсы: Меньше дублирования в [Install] (только graphical-session.target).
Минусы: Нужно знать точные значения XDG_CURRENT_DESKTOP.

[Unit]
PartOf=graphical-session.target
After=graphical-session.target

# Запускать ТОЛЬКО если переменная совпадает (ИЛИ)
ConditionEnvironment=|XDG_CURRENT_DESKTOP=sway
ConditionEnvironment=|XDG_CURRENT_DESKTOP=labwc

[Install]
WantedBy=graphical-session.target

--- ВАРИАНТ 4: Универсальный запуск + Blacklist (Черный список) ---
(Пример: mako через отрицание !)
Плюсы: Запускается везде, кроме указанных тяжелых DE (KDE, GNOME).
Минусы: Может запуститься там, где не ожидается (если забыли добавить DE).

[Unit]
# Знак "!" означает отрицание. Запускать, если ЭТО НЕ KDE и НЕ GNOME.
ConditionEnvironment=!XDG_CURRENT_DESKTOP=KDE
ConditionEnvironment=!XDG_CURRENT_DESKTOP=GNOME

5. ФОНОВЫЕ ДЕМОНЫ И ONESHOT ЗАДАЧИ
================================================================================

--- ВАРИАНТ 5: Фоновый сервис (Type=simple) ---
(Пример: ydotoold - эмулятор ввода)
Используется для демонов, которые работают постоянно.
Type=simple (или exec) — основной процесс не форкается.

[Service]
Type=simple
Restart=always           # Всегда перезапускать при падении
KillMode=process         # Убивать только главный процесс (не всю группу)
TimeoutSec=180           # Ждать 180 сек перед принудительным убийством

[Install]
WantedBy=default.target  # Стандартная цель для пользовательской сессии

--- ВАРИАНТ 6: Пользовательский Oneshot ---
(Пример: Скрипт сканирования Wi-Fi)
Выполняется один раз и завершается. Идеально для скриптов инициализации.

[Service]
Type=oneshot
ExecStart=%h/.local/bin/iwctl-auto-scan.sh  # %h = домашняя директория пользователя
RemainAfterExit=yes                         # (Опционально) Считать сервис активным после выполнения

[Install]
WantedBy=default.target

--- ВАРИАНТ 7: Системный Oneshot (С примерами) ---
Запускается от root при загрузке системы.
Примеры использования:
1. Применение sysctl настроек, которых нет в /etc/sysctl.conf.
2. Исправление прав на файлы перед запуском сервиса.
3. Одноразовая миграция данных.

Пример конфига (/etc/systemd/system/fix-permissions.service):
[Unit]
Description=Fix permissions for app data
After=local-fs.target

[Service]
Type=oneshot
ExecStart=/usr/bin/chown -R appuser:appgroup /var/lib/myapp
ExecStart=/usr/bin/chmod 750 /var/lib/myapp
RemainAfterExit=yes

[Install]
WantedBy=multi-user.target  # Цель для многопользовательской системы (консоль/сервер)

6. УПРАВЛЕНИЕ ПРИОРИТЕТАМИ И ПРАВАМИ (CAPABILITIES)
================================================================================
Иногда обычному приложению нужны особые права (например, менять приоритет CPU).

--- ВАРИАНТ 8: Пользовательский сервис с изменением приоритета ---
(Пример: gamescope)
Пользователь не может просто так повысить приоритет (nice value).
Нужно разрешить это в юните.

[Service]
# Разрешаем процессу менять свой nice в диапазоне от -20 до 19
LimitNICE=-20
# Или использование Nice напрямую (если разрешено политикой)
Nice=-10 
# Для игр часто требуется realtime приоритет (требует настройки limits.conf)
LimitRTPRIO=90

--- ВАРИАНТ 9: Системный сервис с CapabilityBoundingSet ---
(Пример: Сервис, требующий CAP_SYS_NICE)
Вместо запуска от root, даем сервису только конкретную "капабилити" (право).
Это безопаснее, чем полный root.

[Service]
# Ограничиваем набор прав только тем, что нужно
CapabilityBoundingSet=CAP_SYS_NICE
# Делаем эти права доступными для процесса (Ambient)
AmbientCapabilities=CAP_SYS_NICE
# Запускать от обычного пользователя, но с правом на Nice
User=gameuser
Group=gameuser

[Install]
WantedBy=multi-user.target

7. СВОДНАЯ ТАБЛИЦА ДИРЕКТИВ И SLICE
================================================================================

| Директива                 | Значение / Пример                          | Комментарий                              |
|---------------------------|--------------------------------------------|------------------------------------------|
| Slice=                    | background-graphical.slice                 | Ограничение ресурсов (CPU/RAM)           |
|                           | app-graphical.slice                        | Для интерактивных приложений             |
|                           | user-.slice                                | По умолчанию для юзера                   |
| PartOf=                   | graphical-session.target                   | Связь жизненного цикла (стоп -> стоп)    |
| After=                    | network.target                             | Порядок запуска (не гарантия зависимости)|
| ConditionEnvironment=     | |XDG_CURRENT_DESKTOP=sway                 | Условие "ИЛИ" (запуск если совпало одно) |
| ConditionEnvironment=     | !XDG_CURRENT_DESKTOP=GNOME                 | Условие "НЕ" (запуск если не совпало)    |
| WantedBy=                 | default.target                             | Для user-сервисов (вход в сессию)        |
|                           | multi-user.target                          | Для system-сервисов (загрузка ОС)        |
| Type=                     | simple, exec, oneshot                      | Поведение главного процесса              |
| Restart=                  | on-failure, always                         | Политика перезапуска при краше           |
| CapabilityBoundingSet=    | CAP_SYS_NICE                               | Какие права root доступны сервису        |
| AmbientCapabilities=      | CAP_SYS_NICE                               | Передача прав в исполняемый процесс      |

8. ПОШАГОВАЯ ЛОГИКА РАБОТЫ SYSTEMD (АЛГОРИТМ)
================================================================================
1. Чтение конфига: systemd читает файл .service.
2. Проверка условий ([Unit] Condition...): 
   Если условие ложно -> Статус "skipped", запуск отменяется.
3. Проверка зависимостей ([Unit] After/Requires):
   systemd ждет, пока указанные юниты не будут активны.
4. Запуск процесса ([Service] ExecStart):
   В зависимости от Type, systemd решает, когда считать сервис "запущенным".
5. Мониторинг:
   systemd следит за PID. Если процесс упал и Restart=always -> перезапуск.
6. Остановка:
   При выключении или остановке Target, systemd шлет SIGTERM, ждет TimeoutSec, 
   затем шлет SIGKILL.
7. Установка ([Install]):
   При команде 'systemctl enable', systemd создает симлинк в папке 
   ~/.config/systemd/user/<target>.wants/ или /etc/systemd/system/<target>.wants/.

================================================================================
9. ЗАКЛЮЧЕНИЕ И СОВЕТЫ
================================================================================
1. Всегда проверяйте синтаксис: 
   $ systemd-analyze verify ~/.config/systemd/user/myservice.service

2. Для отладки используйте:
   $ systemctl --user status <service>
   $ journalctl --user -u <service> -f

3. Slice (Слайсы):
   - background-graphical.slice: Для уведомлений, баров (чтобы не лагало окно).
   - app-graphical.slice: Для тяжелых приложений.
   - Если не указать Slice, сервис попадет в общий слайс пользователя.

4. Безопасность:
   - Избегайте запуска сервисов от root, если можно обойтись AmbientCapabilities.
   - Используйте ConditionEnvironment, чтобы не запускать Wayland-сервисы в X11/GNOME.

5. [Install] обязателен, если вы хотите, чтобы сервис жил после перезагрузки 
   без ручного запуска.
